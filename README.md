# 课程云产品设计思路

这个产品的设计核心，是围绕 **"课程内容的生产与编排"** ，构建了一套**兼顾效率与灵活性的双轨协作体系**。它尊重内容创作的自然规律——结构规划与内容生产往往是交替进行的，而不是线性的。

---

### 1. 双轨制生产：AI 效率 × 人类判断

产品设计了两条原生路径，而非事后拼接：

- **AI 驱动路径（蓝图导入）**：面向"生成"，让 AI 批量产出结构化的课程骨架，解决从0到1的效率问题。
- **人类驱动路径（Studio 编辑）**：面向"精调"，提供直观的树状编辑器，让教学设计师专注于内容质量、逻辑和细节打磨。

**核心思想**：不是用 AI 替代人，而是让 AI 完成粗加工，人类完成精加工，两条线产出完全兼容的数据。

### 2. 树形结构作为通用语言

定义了 `Program → Course → Phase → Lesson → Scene → Step` 六级树，这不是简单的文件夹层级，而是有**教学语义**的骨架：

- **上四级（专业到课时）** 负责**教学组织**，是人类编辑和 AI 生成的主要操作面。
- **下两级（场景到步骤）** 负责**内容细节**，承载实际操作引导。

这种分层将"课程大纲规划"与"内容细节填充"自然解耦，让不同角色可以聚焦在不同层级。

### 3. 支持双向工作流，破除规划僵局

产品明确拥抱两种极端的创作习惯：

- **场景 1A（自上而下）**：适合体系化规划，先搭完整骨架，再填血肉。符合传统课程设计的"蓝图思维"。
- **场景 1B（自下而上）**：适合内容驱动，先有零散的课时或大纲，再生长出上层结构。承认"计划赶不上变化"，允许边做边想。

**核心思想**：工具不强制用户遵循单一生产流程，每一层节点都是独立资源，可以随时"回头调整结构"，降低了规划焦虑。

### 4. 资源独立性：让内容不被结构锁死

关键设计是每层节点（Program/Course/Lesson）都作为**独立资源**存在，不因上层容器未发布或变更而失效。删除专业不会级联删除课程，只是不展示；课时可以从一个阶段拖到另一个阶段。

这实现了**内容资产的灵活重组**，课程结构就像可以随时重新排列的积木，而非焊死的船体。

### 5. 蓝图作为互通凭证

导入和导出功能不仅仅是文件操作，而是一种**数据流通协议**：

- 蓝图（JSON）是 CLI 和 Studio 的**共同血脉**，保证了 AI 生成的内容可以无缝流入人类编辑流，Studio 精细调整后的结果也可以再次导出，供其他实例或 CLI 使用。
- 冲突处理策略（覆盖/跳过/保留两者）体现了对**人工劳动成果的保护**，避免自动化导入粗暴覆盖已精调的内容。

### 6. 渐进式发布与安全护栏

状态只有 `draft` 和 `published` 两种，足够清晰。配合安全规则：

- 编辑时未保存有标记，防误丢失。
- 删除需确认并提示子级数量，防误操作。
- 发布时若有草稿子内容会提示，确保教学侧不会看到半成品。
- 已发布内容不可直接删除，必须先下架，形成**发布保护**。

### 7. 操作即排序，位置即关系

排序不靠填写数字，而是直接拖拽。这反映了产品设计中的**直觉化操作**思维：可视化树上的物理位置直接体现教学顺序，将复杂的排序管理变为直觉化的拖动行为。

---

**总结**：这个产品的设计思路，是将课程制作视为一个"粗胚成型 → 精细雕刻 → 灵活重组"的循环过程，而非一次性的文档填写任务。它通过 AI 与人类在相同数据模型上的协作，让结构化内容的生产既快、又准、且不易失控。
